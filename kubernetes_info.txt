=================== КАК работает ingress ===================
Схема трафика через ингресс в спрингбут приложение, если указываем имя порта (http), а не номер
order-service.local:80
	  |
	  v
NGINX Ingress Controller
(192.168.1.100:80, т.к. 80 - порт для http)
Что делает:
	Пришел запрос на order-service.local
	путь начинается с /
	отправляю в сервис order-service.local
	на порт с именем http
	  |
	  |
	  |
	  | с помощью Ingress rule
	  |     - host: order-service.local
	  |		  http:
	  |			paths:
	  |			  - path: /
	  |				pathType: Prefix
	  |				backend:
	  |				  service:
	  |					name: order-service
	  |					port:
	  |					  name: http
	  |
	  |
	  | ищет Service port с именем "http"
	  |
	  v
Service:
	...
	ports:
    - port: {{ .Values.service.port }}	# т.е., например, 80
      targetPort: application-port 	#  указывает на порт, который слушает приложение внутри контейнера)
									# NOTE: Очень часто это тот же порт, который указан в EXPOSE, но Kubernetes не читает Dockerfile => не использует EXPOSE при работе.
      protocol: TCP
      name: http
	  |
	  |
	  | ищет Pod port с именем "application-port"
	  |
	  v
Pod (based on Deployment)
	...
	containers:
	  - name: {{ .Chart.Name }}
        ports:
          - name: application-port
            containerPort: {{ .Values.container.port }}	# т.е. 8081
            protocol: TCP
	  |
	  v
Spring Boot (8081)

НО best practice - называть порты одинаково и в сервисе, и в деплойменте, чтобы была простая цепочка: http -> http -> 8081

Итого:
| Уровень          | Значение                 | Где задан                      |
| ---------------- | -------------------------| ------------------------------ |
| Внешний HTTP     | 80                       | Ingress Controller             |
| Service port     | 80                       | service.yaml                   |
| targetPort       | http -> 8081             | service.yaml + deployment.yaml |
| containerPort    | application-port -> 8081 | deployment.yaml                |
| Spring Boot port | 8081                     | application.properties         |


Логика:
Ingress знает только про Service.
Service знает только про Pod.
Pod знает только про приложение.
Каждый слой имеет свой порт.

ПРОФИТ в именах портов - можно изменить внутренний порт контейнера, и не нужно нигде это менять, кроме Deployment-а. т.е. Service не трогаем, например.

Правила:
    имя порта должно соответствовать требованиям Kubernetes:
    только строчные буквы; цифры; -;
    максимум 15 символов.

КАК достучаться до сервиса, имеющего ингресс, с локального хоста.
ПРОБЛЕМА
	При отправке REST запроса получаю ошибку
	java.net.UnknownHostException: Этот хост неизвестен (order-service.local)
ПРИЧИНА:
	Ingress не создает DNS-запись. Он только говорит:
	"Если придет HTTP-запрос с заголовком Host: order-service.local, то направить его в Service order-service."
	Но кто-то еще должен суметь преобразовать имя order-service.local в IP-адрес ingress-контроллера.
	Процесс:
		POST http://order-service.local/api/orders
				│
		Windows пытается найти IP для order-service.local
				│
		DNS отвечает: такого имени нет
				│
		UnknownHostException

	До Kubernetes запрос вообще не дошел.

РЕШЕНИЕ:
	1) Запустить Блокнот от имени администратора
	2) Открыть файл C:\Windows\System32\drivers\etc\hosts
		добавить в низ файла строку
		127.0.0.1 order-service.local
		сохранить и закрыть
	3) Проверить, что DNS работает: в cmd выполнить ping order-service.local

=================== КАК запускается Pod (на примере artemis) ===================
info:
https://chatgpt.com/c/6a50c7d5-b834-83eb-93ea-04b0b6095015

    Жизненный цикл:
    Pod создается
          |
    initContainer #1
          |
    initContainer #2
          |
    ВСЕ успешно завершились
          |
    Запускаются обычные containers
          |
    Pod Ready

NOTE: Ни один обычный контейнер не запустится, пока все initContainer не завершатся успешно.

В случае Artemis:
    Pod стартует
            |
    initContainer
            |
            |  копирует broker.xml
            |  копирует users/roles
            |
    основной контейнер Artemis
            |

    читает уже измененные конфиги
            |
    стартует


Т.е. внутри Pod может быть несколько контейнеров.
initContainer-ы - для подготовки окружения.
    т.е., например,
    скачать дамп БД
    ↓
    распаковать
    ↓
    запустить PostgreSQL

Каждый initContainer - для отдельной логической задачи.

На примере helm/infra/artemis/templates/deployment.yaml

1) Kubernetes создает Pod
2) Kubernetes создает PVC
    volumes:
      - name: data
        persistentVolumeClaim:
          claimName: {{ include "artemis.fullname" . }}-data
2.2) и configmap
      - name: config
        configMap:
          name: {{ include "artemis.fullname" . }}-config
3) Kubernetes монтирует PVC:
    volumeMounts:
      - name: data
        mountPath: /var/lib/artemis-instance
NOTE: во время монтирования папка /var/lib/artemis-instance появляется автоматически как точка монтирования! Но он пустой! Без bin, etc, data, lib
3.2) Kubernetes монтирует config в папку /config
4) Запускается initContainer
    выполняется /opt/activemq-artemis/bin/artemis create /var/lib/artemis-instance
    к-ая создает
        /var/lib/artemis-instance
        ├── bin
        ├── etc
        ├── data
        ├── lib
        ├── log
        └── tmp
4.2) выполняется точечное копирование файлов из /config в /var/lib/artemis-instance/etc/
    типа cp /config/broker.xml /var/lib/artemis-instance/etc/broker.xml
    NOTE: если просто написать типа cp /config /var/lib/artemis-instance/etc

НА ЭТОМ РАБОТА initContainer-а заканчивается! и он переходит в статус

5) запускается обычный container 'artemis'
    к нему монтируется тот самый PVC data в папку /var/lib/artemis-instance
    в этом PVC уже есть все необходимое, приготовленное initContainer-ом

    выставляются порты

    стартует artemis

=================== КАК определить, как построить DNS-имя ===================
	1) выполнить
	kubectl get svc -A
	результат:
		NAMESPACE     NAME              TYPE        CLUSTER-IP      PORT(S)
		infra     artemis           ClusterIP   10.96.125.42    61616/TCP

	2) Полное DNS-имя строится по шаблону:
		<service>.<namespace>.svc.cluster.local
		но <service>.<namespace> тоже сработает

	3) А в application.properties сервиса пишем spring.artemis.broker-url=tcp://artemis.infra:61616 или параметризуем


=================== КАК открыть Web Console artemis-a ===================
	1) kubectl port-forward svc/artemis 8161:8161 -n infra
	2) открыть в браузере http://localhost:8161

=================== Полезные команды ===================
Запуск контейнера из имаджа artemis (можно без -rm):
    kubectl run artemis-debug -n infra --rm -it --image=apache/activemq-artemis:2.44.0 --command -- sh

Посмотреть логи:
    kubectl logs artemis-debug -n infra

Посмотреть логи контейнера init-artemis в поде artemis-7ccbf9f5fb-66nc5:
	kubectl logs artemis-7ccbf9f5fb-66nc5 -n infra -c init-artemis

Посмотреть итоговую конфигурацию поды artemis-7ccbf9f5fb-66nc5:
    kubectl describe pod artemis-7ccbf9f5fb-66nc5 -n infra

Временно выставить порт для доступа/отладки приложения:
    kubectl port-forward service/artemis 8161:8161 -n infra

=================== Установка с помощью Helm ===================
Helm:
info: https://chatgpt.com/c/6a50951b-f3b8-83ed-bf35-6a89f8737117

1) создаем в корне проекта папку helm
2) в helm папке выполняем команду
        helm create order-service
    она создает структуру папок и файлов helm chart-а, которыую при желании можно потом поменять.

Если хотим деплоить в кубер вручную, то выполняем:
    helm install order-service ./order-service
Если поменяли что-то в helm chart-е, то выполняем:
    helm upgrade order-service ./order-service
Чтобы удалить все сущности сервиса:
    helm uninstall order-service

Как проверить, что конфигурация helm корректна:
1) проверка синтаксиса:
    helm lint ./helm/infra/artemis
2) проверка результирующих кубер файлов:
    helm template artemis ./helm/infra/artemis -f ./helm/infra/artemis/values.yaml
